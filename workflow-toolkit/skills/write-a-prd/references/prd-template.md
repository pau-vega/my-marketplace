## Problem Statement

The problem from the user's perspective. What is broken, missing, or painful.

## Solution

The proposed solution from the user's perspective. What changes and why.

## User Stories

A numbered list of user stories covering all aspects of the feature:

1. As a [actor], I want [feature], so that [benefit]
2. ...

Make this list extensive — cover the happy path, error cases, edge cases, and administrative workflows. 10-30 stories is normal for a meaningful feature.

## Implementation Decisions

Decisions made during the interview:

- Modules to build or modify
- Interfaces and boundaries between modules
- Technical clarifications
- Architectural decisions
- Schema changes
- API contracts
- Specific interactions and flows

Do NOT include specific file paths or code snippets — they become outdated quickly.

## Testing Decisions

- Description of what makes a good test (test external behavior, not implementation details)
- Which modules will be tested
- Prior art for tests (similar existing tests in the codebase)

## Out of Scope

What is explicitly not part of this work. Be specific — vague exclusions invite scope creep.

## Further Notes

Any additional context, constraints, or considerations.
